## COMPREHENSIVE SKILL EXTRACTION & MATCHING RULES:
1. CANDIDATE SKILL EXTRACTION (MANDATORY 100% COVERAGE):
   - You MUST carefully scan the entire Resume Text AND the Candidate Form Data (partnerReportedSkills, writeup).
   - Extract ALL skills, technologies, frameworks, tools, programming languages, libraries, databases, methodologies, functional competencies, and soft skills listed.
   - Do NOT summarize, truncate, or omit skills. If there are 30 skills listed, extract all 30. Return the full comprehensive list in the `candidateProfile.skills` array. Omissions are a failure of analysis.

2. INPUT FORMAT & TIERS MAPPING:
   - Identify the complete set of required/must-have and preferred/should-have/nice-to-have skills requested in the Job Description.
   - If the Job Description is raw text: look for "MUST-HAVE Skills" (map to required/must-have) and "PREFERRED Skills" (map to preferred/nice-to-have).
   - If the Job Description is JSON: map "skills.mustHave" to required/must-have, "skills.shouldHave" to should-have/preferred, and "skills.niceToHave" to nice-to-have/preferred.

3. STRICT SKILLS MATCHING PROTOCOL:
   - A skill in the Job Description is either MATCHED or MISSING. There is NO middle ground or fuzzy assumption.
   - To be classified as MATCHED, the candidate MUST have the exact skill name, an identical alias (from the list below), or be a direct superset (e.g. TypeScript for JavaScript).
   - If the candidate only has a related skill but not the requested skill itself, it MUST NOT be considered a match. You MUST classify it as MISSING.
   - For example:
     - If the JD requires "Node.js", and the candidate only has "JavaScript", "Node.js" is MISSING.
     - If the JD requires "React", and the candidate only has "HTML/CSS" or "Vue" or "Angular", "React" is MISSING.
     - If the JD requires "AWS", and the candidate only has "Cloud Computing" (but no specific AWS services or AWS mentioned), "AWS" is MISSING.
     - If the JD requires "Python", and the candidate only has "C++" or "Java", "Python" is MISSING.
   - NEVER assume, extrapolate, or imply a skill. If it is not explicitly listed in their resume text or form data, it is MISSING. No exceptions.

4. MATCHING & MISSING SKILLS CLASSIFICATION:
   - For EVERY single skill identified in the Job Description, you MUST check if it matches any skill extracted for the candidate (using the STRICT MATCHING PROTOCOL).
   - If it matches: place it in the appropriate matched list:
     - Required/Must-have matches MUST go into `rankingSignals.mustHaveSkillsMatched`.
     - Should-have matches MUST go into `rankingSignals.shouldHaveSkillsMatched`.
     - Nice-to-have/Preferred matches MUST go into `rankingSignals.preferredSkillsMatched` or `rankingSignals.niceToHaveSkillsMatched`.
   - If it does NOT match: you MUST place it in the missing list:
     - Required/Must-have missing skills MUST go into `rankingSignals.mustHaveSkillsMissing`.
   - Every single skill in the Job Description MUST be classified as either matched or missing. Do NOT skip any skills. If a skill in the Job Description is missing from the candidate's resume/form data, it is MANDATORY that it appears in the missing skills list.

## COMPONENT SCORING (Each 0-100):

### 1. SKILLS MATCH (Weight: 30%) — Tier-weighted:
mustHaveMatched   = count of mustHave skills found in resume (using alias rules)
shouldHaveMatched = count of shouldHave skills found
niceToHaveMatched = count of niceToHave skills found

skillsMatch = (mustHaveMatched/MAX(mustHaveTotal,1)*70) + (shouldHaveMatched/MAX(shouldHaveTotal,1)*25) + (niceToHaveMatched/MAX(niceToHaveTotal,1)*5)

IF mustHaveMatched/mustHaveTotal is below 0.70: cap skillsMatch at 50.
skillCoveragePercent = (mustHaveMatched+shouldHaveMatched)/MAX(mustHaveTotal+shouldHaveTotal,1)*100

### 2. EXPERIENCE MATCH (Weight: 20%):
Use LOWER of (formData.totalExperience, actualExperienceFromResume).
candidate >= maxRequired: 100
within range (min to max): 100
1-2 below min: 70
2-3 below min: 40
3+ below min: 20

### 3. DOMAIN MATCH (Weight: 15%):
EXACT same role type: 100
RELATED adjacent field: 70
UNRELATED: 20

### 4. EDUCATION MATCH (Weight: 10%):
Hierarchy: 10th Pass < 12th Pass < Diploma < Bachelor's < Master's/MBA < PhD
Exceeds: 100 | Exact: 90 | One level below: 70 | Two levels below: 30 | Three+ below: 0

### 5. SALARY FIT (Weight: 10%):
expected <= budgetMax: 100
1-10% over: 80
11-20% over: 60
21-30% over: 40
more than 30% over: 0
missing data: 70 (neutral, cannot assess)

### 6. LOCATION MATCH (Weight: 5%):
Remote job OR same city: 100
Same state or within 100km: 80
Different state, candidate open to relocation: 60
Different region, no relocation indicator: 20

### 7. NOTICE PERIOD FIT (Weight: 5%):
0-15 days: 100
16-30 days: 90
31-45 days: 80
46-60 days: 70
61-90 days: 50
more than 90 days: 30

### 8. STABILITY SCORE (Weight: 5%):
avg tenure = total months in all jobs / number of jobs
36+ months per job: 100
24-35 months: 80
18-23 months: 60
12-17 months: 40
6-11 months: 20
less than 6 months: 0

## WEIGHTED SCORE FORMULA:
weightedScore = (skillsMatch*0.30) + (experienceMatch*0.20) + (domainMatch*0.15) + (educationMatch*0.10) + (salaryFit*0.10) + (locationMatch*0.05) + (noticePeriodFit*0.05) + (stabilityScore*0.05)
Round to nearest integer.

---

## RISK PENALTIES — Hard caps enforced:
careerGapPenalty (max 5): career gap more than 6 months = -5
jobHopperPenalty (max 8): avg tenure less than 12 months = -8
domainMismatchPenalty (max 10): UNRELATED domain = -10
experienceDiscrepancyPenalty (max 7): MAJOR_DIFF detected = -7
salaryOverBudgetPenalty (max 5): expected more than 30% over budget = -5

riskPenalty = MIN(sum of all penalties, 25)   <- HARD CAP, never exceed 25
finalAdjustedScore = MAX(0, weightedScore - riskPenalty)   <- FLOOR at 0

---

## PRIORITY SCORE:
priorityScore = (mustHaveMatchedCount/MAX(mustHaveTotal,1)*40) + (finalAdjustedScore*0.40) + (noticePeriodFit*0.10) + (salaryWithinBudget ? 10 : 0)
Round to integer. Range 0-100.
